00 — Overview
What is HeadyBot?

HeadyBot is the autonomous agent framework powering the Heady ecosystem. It orchestrates a swarm of specialized AI micro-agents — called bees — across 17 logical swarm clusters. Each bee is a stateless, sandboxed task worker equipped with access to the 47-tool MCP array and a localized shadow memory via Redis.

The octagonal geometry of Layer 7 represents the eight symmetrical axes of automation: spawning, execution, reporting, retiring, consensus, scheduling, orchestration, and self-optimization. HeadyBot continuously monitors swarm health, auto-tunes concurrency via heady-eval, and routes inference pressure to the most capable available bee.

All agent counts, concurrency limits, and pipeline stage counts derive from the Fibonacci sequence: 17 swarm types (prime near fib), 89 bee types (fib(11)), and pipeline stages of 8, 13, and 21 (fib(6), fib(7), fib(8)).

17 Swarm Types

Logical agent clusters organized by capability domain. Each swarm has a primary coordinator bee and up to fib(7)=13 worker bees running concurrently.

89 Bee Types

Micro-agent specializations mapped to fib(11)=89. Each bee type has a defined model tier (Premium/Standard/Fast), tool permissions, and memory scope.

DAG Execution

Tasks are represented as Directed Acyclic Graphs. The conductor resolves dependency order and dispatches subtasks in parallel when the DAG topology allows.

Cron Scheduler

Fibonacci-interval cron jobs trigger recurring workflows: health checks at fib(7)=13 min, memory consolidation at fib(9)=34 min, system audits daily.

Swarm Consensus

Multi-bee decisions use a CSL-weighted voting protocol. No action proceeds unless the aggregate confidence score exceeds the INJECT threshold (0.718).

φ Concurrency

Thread pool sizes follow the Fibonacci sequence: 2, 3, 5, 8, 13. The auto-tuner scales within these bounds based on compute pressure and latency deltas.

01.1 — Architecture
HeadyBee Lifecycle

Every bee agent follows a four-phase lifecycle managed by the heady-conductor service. Bees are stateless during execution — they receive task context from the conductor, access tools via MCP, read shadow memory from Redis, and return a structured result before retiring.

🥚
SPAWN
Conductor instantiates bee with task context, model tier, tool permissions bitmask, and Redis session key.
⚡
EXECUTE
Bee processes task through HCFP pipeline. Calls MCP tools as needed. Vector context loaded from Qdrant via shadow memory.
📡
REPORT
Bee returns structured result with CSL confidence score, tool call trace, and token usage. Conductor updates task DAG status.
💤
RETIRE
Bee state written to Shadow Memory (Redis). Worker thread released back to φ-scaled pool. Bee instance dissolved.

01.3 — Architecture
Heady Conductor

The heady-conductor service is the central orchestration layer — the hive mind that coordinates all swarm activity. It maintains the global task queue, manages the DAG execution graph, tracks bee availability across swarms, and auto-tunes concurrency via heady-eval feedback loops.

The conductor exposes a gRPC-compatible internal API consumed by the pipeline engine and the auto-success engine. External automation requests enter via the heady-manager gateway at /api/orchestration.

Conductor Architecture: The conductor is a stateful Cloud Run service with a dedicated Redis connection pool (fib(5)=5 connections minimum, scaling to fib(7)=13 under load). It persists swarm topology, task DAG state, and bee availability maps in Redis with a 55-day TTL (Fibonacci 10th term).

01.4 — Architecture
DAG-Based Task Distribution

HeadyBot represents every multi-step workflow as a Directed Acyclic Graph (DAG). Nodes are individual bee tasks; edges are dependency relationships. The conductor's DAG executor resolves the graph topologically and dispatches all leaf nodes (tasks with no unresolved dependencies) in parallel, up to the active swarm's φ-concurrency limit.

CSL confidence weights gate DAG progression: if a node's bee returns a confidence score below INCLUDE (0.382), the DAG halts at that node and a SENTINEL_SWARM bee is spawned to investigate before execution resumes.

DAG φ Scaling Rules: Maximum DAG depth = fib(7) = 13 steps. Maximum parallel fan-out per level = fib(5) = 5 bees. Maximum total nodes per DAG = fib(10) = 55. These bounds prevent unbounded recursion while enabling complex multi-agent workflows.

02.2 — Features
Swarm Consensus Protocol

When a high-stakes decision requires multiple perspectives, the conductor spawns a consensus session across 3–5 bees from the relevant swarm. Each bee independently evaluates the problem and returns a result with a CSL confidence score. The conductor aggregates scores using a φ-weighted average to determine the final action.

The consensus threshold is the INJECT gate (0.718). If the aggregate CSL score falls below this threshold, the workflow pauses and escalates to a Brain-tier bee for meta-review before proceeding.

// Consensus aggregation — weighted CSL average function aggregateConsensus(results: BeeResult[]): number { const PHI = 1.618033988749895; const weights = results.map((_, i) => 1 / (PHI ** i)); const weightSum = weights.reduce((a, b) => a + b, 0); return results.reduce((acc, r, i) => acc + (r.cslScore * weights[i]) / weightSum, 0 ); } // INJECT gate: 0.718 — aggregate must exceed to proceed const CONSENSUS_THRESHOLD = 0.718;

Triggered on CMS update. Runs SEO analysis → translation → publish via content-bee chain. Supports 5 parallel language targets.

Deployment Flow

Triggered on main branch push. Runs tests → build → Cloud Run promotion via devops-bee + deployment-bee. Rollback on CSL confidence drop.

03.1 — Getting Started
Quickstart

Submit a workflow to HeadyBot via the orchestration API endpoint. Authenticate with a JWT obtained from the heady-manager auth flow, then POST a workflow definition to /api/orchestration.

# Submit a single-bee task curl -X POST https://api.headysystems.com/api/orchestration \ -H "Authorization: Bearer $JWT" \ -H "Content-Type: application/json" \ -d '{ "workflow": { "name": "Quick Research Task", "steps": [{ "id": "s1", "bee": "researcher-bee", "tools": ["analyze", "patterns"], "input": { "objective": "Summarize recent Heady architecture changes" } }] } }' # Response { "workflowId": "wf_01j8...", "status": "QUEUED", "estimatedMs": 11090 }
03.2 — Getting Started
Spawn a Bee via SDK

The @headybot/sdk package provides a typed interface for spawning bees and monitoring workflow progress from Node.js or edge environments.

import { HeadyBot } from '@headybot/sdk'; const bot = new HeadyBot({ apiKey: process.env.HEADY_API_KEY }); // Spawn a single bee const result = await bot.spawn({ bee: 'researcher-bee', objective: 'Audit the Heady security posture for Q2 2025', tools: ['risks', 'analyze', 'deep_scan'], timeout: 11090, // TIMING.NORMAL }); console.log(result.cslScore); // 0.882 → HIGH confidence console.log(result.output); // Structured research report
